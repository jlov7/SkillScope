# Release-ready v1 Design

## 1) Product Scope and UX Principles

SkillScope v1 stays publicly positioned as a research-only toolkit, but the product experience will behave like a production-ready offering. The primary audience is developers integrating Skill instrumentation, with a secondary audience of product/ops stakeholders who need summaries and demos. The v1 UX should feel confident, crisp, and practical: emphasize “what you can do now” (instrument, analyze, demo) rather than aspirational promises. The web UI will be a lightweight companion to the CLI, not a replacement, and must offer a coherent, end-to-end story: onboarding to analysis to sharing results. The onboarding flow should be explicit about “client-side, no uploads” to preserve trust and privacy, and it should clearly state the research-only positioning without undermining confidence in the tooling. Empty states should be instructional, with progressive disclosure for advanced options (format detection, optional overrides). The demo experience should be first-class: a guided path that loads bundled sample data and highlights what metrics mean. Copy should be concise, with primary actions surfaced above the fold and supporting details available via tooltips or secondary panels. Accessibility is a baseline requirement: keyboard-first navigation, visible focus states, and unambiguous labels for inputs. Visual style should feel intentional and innovative, while still aligned with developer tooling expectations—clean typography, data-forward layouts, and subtle motion to guide attention without distraction.

## 2) Web UI Architecture and Core Flows

The v1 web UI is a static Next.js app deployed to Vercel, with client-side analysis only. This keeps security and deployment risk low while delivering a polished UX. Core flows:

- Onboarding: first-run screen with a short overview, privacy notice, and a “Load demo” or “Upload data” split. Persist a “seen onboarding” flag in localStorage.
- Analyze: upload JSON/JSONL/Anthropic JSON in-browser, parse, normalize, and summarize, then display KPI cards, skill-level tables, and file/model breakdowns. Provide clear error states with remediation tips.
- Demo: one-click analysis using bundled sample data (derived from the CLI demo) to showcase dashboards without user data.
- Help: an embedded help page with minimal docs plus contextual tooltips on the Analyze screen.
  The UI should be structured with a minimal design system: typography scale, color tokens, and reusable components for KPI cards, tables, alerts, and empty states. Avoid unnecessary re-renders by memoizing expensive analysis steps and using file-size limits to protect the browser. Parsing should be robust: support JSON arrays, JSON objects, JSONL, and Anthropic message payloads. Large files should be handled defensively, with a clear size cap and a sample-size warning. The UI should surface “research-only” in a persistent but unobtrusive way, such as a footer badge and a short note near the onboarding call-to-action.

## 3) Data Model, Analysis Logic, and Quality Gates

Client-side analysis mirrors the CLI’s normalization and summarization logic. The analyzer should normalize events to a canonical schema (attrs, skill metadata, token usage), then compute totals and per-skill aggregates: calls, policy rate, tokens, top files, models, and progressive levels. The JSON output should match the CLI’s `--format json` as closely as practical to maintain parity. This logic is critical and must have unit tests (Vitest) with fixtures derived from the existing demo events and synthetic edge cases. Key UI flows—onboarding, demo analysis, error state—need test coverage via browser-level tests (Playwright) or component tests with React Testing Library, whichever provides the clearest coverage with minimal overhead. Linting, typechecking, and build must be added to CI for both Python and Web UI. Documentation updates must include web setup, run/test/build commands, and Vercel deployment notes. Security hygiene: no secrets in the repo, validation for file size and content, and no network requests by default. Performance basics: the analyzer should be written to avoid repeated parsing, and results should be memoized. The final release gate is a checklist-driven verification: lint, typecheck, tests, build, and a manual pass through the onboarding + demo flow.
